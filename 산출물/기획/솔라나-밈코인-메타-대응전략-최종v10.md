# FYRST 솔라나 밈코인 메타 대응 전략 (최종 개발 지침서 v10 - Raydium 로컬 테스트 완벽 가이드)

> **문서 목적:** AI 에이전트가 이 문서를 읽고 즉시 온체인 프로그램(Rust), 백엔드, 프론트엔드를 에러 없이 안전하게 수정할 수 있도록 구체적인 구현 로직을 정의합니다. 
> **🚨 핵심 업데이트:** Helius 의존성으로 인해 막막했던 **'로컬 포크 환경에서의 Raydium 졸업 테스트'**를 완벽하게 수행할 수 있는 2가지 상세 전략을 추가했습니다.

---

## 1. 에스크로 담보금 및 마감시간(Deadline) 자율화

*   **상태 구조체:** `EscrowVault`에 `deadline_timestamp: i64` 추가.
*   **생성:** 런칭 시 `duration_seconds`를 받아 마감시간 설정.
*   **하한선:** `MIN_COLLATERAL` = 0.1 SOL.

## 2. 졸업(Graduation) 및 환불(Refund) 로직의 완벽한 분리

*   **`release_escrow`:** `require!(curve.graduated == true)` (졸업 성공 시 즉시 회수)
*   **`process_refund`:** `require!(curve.graduated == false)` AND `require!(now >= escrow.deadline_timestamp)` (마감시간 종료 및 실패 시 유저 분배)

## 3. 프레셔 스코어 (Pressure Score) 및 전광판 UI

*   **공식:** `(에스크로 SOL / max(남은 시간(초), 1)) * 100`
*   **아키텍처:** 백엔드는 정적 데이터만 제공, 프론트엔드에서 1초마다 재계산 및 정렬(Re-sorting)하여 서버 부하 제로화.

## 4. 수수료 클레임 점진적 잠금 해제 (Progressive Fee Unlock)

*   **공식:** `unlocked_fees = (total_deployer_fees * max_reserve_reached) / GRADUATION_THRESHOLD`
*   **보안:** `u128` 캐스팅으로 Overflow 방지.

## 5. 온체인 데이터 동기화 (Helius Webhook)

*   **프로덕션/데브넷:** 프론트엔드 API 호출을 믿지 않고, **Helius Webhook**을 통해 들어온 온체인 트랜잭션 데이터만 파싱하여 DB(`Trade`, `Token`)를 업데이트합니다. (봇 거래 누락 방지)

---

## 6. 🚨 [CRITICAL] Raydium 로컬 포크 테스트 상세 가이드

유저님의 정확한 지적대로, 로컬 포크(Localnet) 환경에서는 Helius Webhook이 동작하지 않습니다. 하지만 로컬 환경은 **"오직 내가 발생시킨 트랜잭션만 존재한다"**는 엄청난 장점이 있습니다. 이를 활용하여 Raydium 졸업을 테스트하는 2가지 완벽한 방법을 제시합니다.

### 🧪 방법 A: CLI 스크립트 기반 스마트 컨트랙트 테스트 (가장 추천)
Raydium CPI(Cross-Program Invocation) 연동은 본질적으로 **'스마트 컨트랙트 간의 통신'**입니다. 프론트엔드나 DB(Helius)가 굳이 개입할 필요가 없습니다.

1.  **환경 세팅:**
    ```bash
    solana-test-validator --url https://api.mainnet-beta.solana.com --reset
    solana airdrop 10000
    anchor deploy --provider.cluster localnet
    ```
2.  **테스트 스크립트 작성 (`scripts/test-raydium-graduation.ts`):**
    *   이 스크립트는 백엔드/DB를 완전히 무시하고 오직 온체인 프로그램하고만 통신합니다.
    *   **Step 1:** `init_protocol` 및 `create_escrow` + `init_bonding_curve` 호출.
    *   **Step 2:** `buy_tokens`를 호출하되, 한 번에 **85 SOL**어치를 매수하여 강제로 `graduated = true` 상태를 만듭니다.
    *   **Step 3:** `graduate` (또는 Raydium CPI를 호출하는 인스트럭션)를 호출합니다.
    *   **Step 4:** 스크립트 내에서 `@solana/spl-token`을 사용해 Raydium CPMM 풀이 정상적으로 생성되었는지, 그리고 **LP 토큰이 영구 소각(Burn) 주소로 이동했는지** 검증(Assert)합니다.
    *   **Step 5:** `release_escrow`를 호출해 데브가 담보금을 무사히 돌려받는지 확인합니다.

### 🖥️ 방법 B: E2E UI 테스트를 위한 "로컬 전용 RPC 리스너" (Fallback)
만약 프론트엔드에서 "🚀 Raydium으로 마이그레이션 중..." 화면이 잘 뜨는지, 차트가 잘 바뀌는지 눈으로 직접 보고 싶다면 백엔드에 **로컬 전용 우회로**를 만들어야 합니다.

1.  **백엔드 환경변수 설정:**
    *   `.env`에 `LOCAL_TEST_MODE=true` 설정.
2.  **로컬 전용 RPC 리스너 가동 (`service/src/index.ts`):**
    *   백엔드 서버가 켜질 때, `LOCAL_TEST_MODE`가 켜져 있으면 Helius 웹훅을 기다리지 않고 **Solana Native RPC의 `connection.onLogs`**를 실행합니다.
    ```typescript
    if (process.env.LOCAL_TEST_MODE === "true") {
      connection.onLogs(PROGRAM_ID, (logs, ctx) => {
        // 로컬 환경은 트래픽이 나 혼자뿐이므로 onLogs가 절대 누락되지 않음.
        // 여기서 트랜잭션을 파싱해서 Helius 웹훅이 온 것처럼 DB를 업데이트함.
        parseAndSyncTradeToDB(logs.signature); 
      }, "confirmed");
    }
    ```
3.  **UI 테스트 진행:**
    *   웹 브라우저를 켜고 런칭 -> 매수(85 SOL)를 진행합니다.
    *   로컬 RPC 리스너가 이를 감지해 DB를 업데이트하고 소켓을 쏩니다.
    *   프론트엔드가 `graduated: true` 소켓 이벤트를 받으면 즉시 거래 버튼을 비활성화하고 **"Raydium 마이그레이션 중"** UI를 띄웁니다.

### 🌉 Raydium CPMM 연동 시 주의사항 (Agent Action)
*   Raydium V4(Amm) 대신 최신 **Raydium CPMM (Core)** 프로그램을 타겟으로 CPI를 작성하는 것이 가스비(CU)와 로직 면에서 훨씬 깔끔합니다.
*   졸업 시 본딩 커브 PDA에 모인 85 SOL과, 커브 PDA에 남아있는 잔여 SPL 토큰 전량을 Raydium CPMM `initialize` 인스트럭션에 밀어 넣어야 합니다.
*   풀 생성 후 반환받는 **LP 토큰(유동성 증명 토큰)은 반드시 온체인 상에서 소각(Burn)** 처리하여 러그풀이 불가능함을 온체인으로 증명해야 합니다.
