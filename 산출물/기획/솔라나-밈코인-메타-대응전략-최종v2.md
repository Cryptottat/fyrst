# FYRST 솔라나 밈코인 메타 대응 전략 (최종 개발 지침서 v2)

> **문서 목적:** AI 에이전트가 이 문서를 읽고 즉시 온체인 프로그램(Rust), 백엔드, 프론트엔드를 에러 없이 안전하게 수정할 수 있도록 구체적인 구현 로직과 수학적 공식을 정의합니다.

---

## 1. 에스크로 담보금 및 마감시간(Deadline) 자율화

배포자(Dev)가 런칭 시 담보금뿐만 아니라 **'졸업 마감시간(Deadline)'**도 스스로 선택하게 합니다. 마감시간을 짧게 잡을수록(예: 1시간) 실패 리스크가 크지만, 그만큼 플랫폼 내에서 엄청난 노출도(Hype)를 얻게 됩니다.

### 🛠️ 온체인 스마트 컨트랙트 수정 지침 (`programs/fyrst/src/`)
*   **상태 구조체 변경 (`state.rs`의 `EscrowVault`):**
    *   기존 `created_at: i64` 외에 **`deadline_timestamp: i64`** 필드를 추가합니다.
*   **`create_escrow` 인스트럭션 (`escrow.rs`):**
    *   클라이언트로부터 `duration_seconds` (예: 1시간=3600, 24시간=86400)를 인자로 받습니다.
    *   `escrow.deadline_timestamp = Clock::get()?.unix_timestamp + duration_seconds;` 로 저장합니다.
    *   *보안:* `duration_seconds`는 최소 1시간(3600) ~ 최대 7일(604800) 범위로 제한(`require!`)합니다.
*   **담보금 하한선 수정 (`constants.rs`):**
    *   `MIN_COLLATERAL`을 1 SOL에서 **0.1 SOL (100,000,000 lamports)** 로 하향 조정합니다.

## 2. 졸업(Graduation) 및 환불(Refund) 로직의 완벽한 분리

"러그풀"이라는 주관적 단어를 시스템에서 완전히 삭제하고, 오직 **"졸업 여부"**와 **"마감시간 초과 여부"**만으로 담보금의 향방을 결정합니다.

### 🛠️ 온체인 로직 수정 지침
*   **`release_escrow` (배포자의 담보금 회수 - `escrow.rs`):**
    *   조건: **`require!(curve.graduated == true)`**
    *   *설명:* 마감시간이 지났든 안 지났든, **졸업을 성공했다면 무조건** 데브는 담보금을 즉시 돌려받을 수 있습니다.
*   **`process_refund` (유저의 보증금 사냥 - `refund.rs`):**
    *   조건 1: **`require!(curve.graduated == false)`** (졸업 실패)
    *   조건 2: **`require!(now >= escrow.deadline_timestamp)`** (마감시간 지남)
    *   *설명:* 데브가 지정한 마감시간이 끝났는데도 졸업을 못 했다면, 홀더들이 데브의 담보금을 N빵해서 가져갑니다.
*   **졸업 처리 안정성 (`bonding_curve.rs`의 `buy_tokens`):**
    *   현재 `reserve_balance >= graduation_threshold` 일 때 `curve.graduated = true`가 되는 로직이 있습니다. 이 부분이 에러 없이 작동하도록 수학적 오버플로우(`checked_add`)를 철저히 확인해야 합니다.

## 3. 프레셔 스코어 (Pressure Score) 및 전광판 UI (Marquee)

짧은 시간 안에 많은 돈이 걸려 있을수록 도파민이 폭발합니다. 이를 수치화한 **Pressure Score(압박 점수)**를 도입하고, 웹 UI를 역동적으로 개편합니다.

### 🧮 Pressure Score 계산식 (백엔드/프론트엔드)
*   **공식:** `(에스크로 SOL) / max(남은 시간(분), 1)`
*   *예시:* 10 SOL이 걸려있는데 마감까지 5분 남았다면? Score = 10 / 5 = **2.0** (초고위험/초고수익 임박)
*   *예시:* 1 SOL이 걸려있고 마감까지 60분 남았다면? Score = 1 / 60 = **0.016** (안정적)

### 🖥️ 프론트엔드 UI/UX 구현 지침 (`web/`)
1.  **상단 전광판 (Top Marquee - 좌에서 우로 흐름):**
    *   **🔥 DEADLINE APPROACHING:** 마감 시간이 1시간 이내로 남은 토큰들을 남은 시간 순(오름차순)으로 보여줍니다. (예: `$DOGE - 00:15:32 남음!`)
2.  **하단 전광판 (Bottom Marquee - 우에서 좌로 흐름):**
    *   **💎 TOP BOUNTY (에스크로 높은 순):** 담보금이 가장 많이 걸린 토큰들을 순위와 함께 보여줍니다. (예: `1위 $PEPE (10 SOL) | 2위 $CAT (5 SOL)`)
3.  **신규 페이지 (`/bounty` 또는 `/pressure`):**
    *   위에서 정의한 **Pressure Score**가 가장 높은 순서대로 정렬된 리스트를 제공합니다.
    *   "실패 직전의 꿀통(Bounty) 사냥"을 노리는 하드코어 유저들을 위한 전용 페이지입니다.
4.  **대시보드 정렬 옵션 추가 (`/dashboard`):**
    *   기존 정렬(최신순, 마켓캡순)에 **'마감시간 임박순(Deadline)'**과 **'담보금 높은순(Escrow)'** 필터를 추가합니다.

## 4. 수수료 클레임 점진적 잠금 해제 (Progressive Fee Unlock)

배포자가 워시 트레이딩(자전거래)으로 수수료만 빼먹고 도망가는 것을 방지하기 위해, **"본딩 커브가 도달한 최대 진행률(Max Progress)"**에 비례해서만 수수료를 클레임할 수 있게 만듭니다.

### 🛠️ 온체인 스마트 컨트랙트 수정 지침 (`state.rs`, `protocol.rs`)
*   **상태 구조체 변경 (`BondingCurve`):**
    *   `claimable_fees` 필드를 삭제하고 아래 3개 필드로 대체합니다.
        1.  `max_reserve_reached: u64` (역대 최대 리저브 잔액)
        2.  `total_deployer_fees: u64` (배포자 몫 총 누적 수수료)
        3.  `claimed_deployer_fees: u64` (배포자가 이미 가져간 수수료)
*   **`buy_tokens` / `sell_tokens`:**
    *   거래 발생 시 `total_deployer_fees`를 누적하고, `max_reserve_reached`를 갱신합니다. (단, 매도 시에는 `max_reserve_reached`를 깎지 않음)
*   **`claim_fees` 로직:**
    *   `unlocked_fees = (total_deployer_fees * max_reserve_reached) / GRADUATION_THRESHOLD`
    *   `amount_to_claim = unlocked_fees - claimed_deployer_fees`
    *   이 공식으로 계산된 금액만 배포자가 클레임할 수 있도록 제한합니다. (나눗셈 시 Zero Division 및 Overflow 주의: `u128` 캐스팅 필수)

---

## 5. 개발 에이전트 주의사항 (Critical Warnings)

1.  **Zero Division 방지:** Pressure Score 계산 시 남은 시간이 0 이하가 될 경우 무한대(Infinity) 에러가 나지 않도록 `Math.max(남은 분, 1)` 처리를 반드시 해야 합니다.
2.  **Rust Math Overflow:** 온체인에서 `unlocked_fees` 계산 시 `total_deployer_fees * max_reserve_reached`가 `u64` 범위를 초과할 수 있으므로 반드시 `u128`로 캐스팅하여 계산 후 `u64`로 다운캐스팅해야 합니다.
3.  **졸업(Graduation) 상태 동기화:** 온체인에서 `curve.graduated = true`가 되는 순간, 백엔드 DB(`Token` 모델)와 프론트엔드 소켓(`trade:executed` 이벤트)에도 이 상태가 즉시 반영되어 UI에 폭죽이 터지거나 상태가 변경되도록 연결 고리를 확인하세요.
4.  **마이그레이션:** 기존에 배포된 테스트 컨트랙트 구조체가 변경되므로, 로컬/데브넷 환경에서 기존 계정 데이터 충돌이 날 수 있습니다. `anchor clean` 후 재배포를 권장합니다.
