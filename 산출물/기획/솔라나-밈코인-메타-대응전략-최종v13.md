# FYRST 솔라나 밈코인 메타 대응 전략 (최종 개발 지침서 v13 - Full Version)

> **문서 목적:** AI 에이전트가 이 문서를 읽고 즉시 온체인 프로그램(Rust), 백엔드, 프론트엔드를 에러 없이 안전하게 수정할 수 있도록 구체적인 구현 로직과 수학적 공식을 상세히 정의합니다. 이 문서는 이전의 모든 기획과 아키텍처 결정을 통합한 최종 버전입니다.

## 1. 에스크로 담보금 및 마감시간(Deadline) 자율화

배포자(Dev)가 런칭 시 담보금뿐만 아니라 **'졸업 마감시간(Deadline)'**도 스스로 선택하게 합니다. 마감시간을 짧게 잡을수록(예: 1시간) 실패 리스크가 크지만, 그만큼 플랫폼 내에서 엄청난 노출도(Pressure Score)를 얻게 됩니다.

### 🛠️ 구현 지침 (Agent Action)
*   **온체인 상수 수정 (`constants.rs`):**
    *   `MIN_COLLATERAL`을 1 SOL에서 **0.1 SOL (100,000,000 lamports)** 로 하향 조정합니다.
*   **상태 구조체 변경 (`state.rs`의 `EscrowVault`):**
    *   기존 `created_at: i64` 외에 **`deadline_timestamp: i64`** 필드를 추가합니다.
*   **`create_escrow` 인스트럭션 (`escrow.rs`):**
    *   클라이언트로부터 `duration_seconds` (예: 1시간=3600, 24시간=86400)를 인자로 받습니다.
    *   `escrow.deadline_timestamp = Clock::get()?.unix_timestamp + duration_seconds;` 로 저장합니다.
    *   *보안:* `duration_seconds`는 최소 1시간(3600) ~ 최대 7일(604800) 범위로 제한(`require!`)합니다.
*   **백엔드 티어 로직 수정 (`service/src/services/escrow.ts` 등):**
    *   담보금 구간별 티어를 다음과 같이 재정의합니다.
        *   `0.1 SOL` 이상 ~ 0.5 미만: **Iron**
        *   `0.5 SOL` 이상 ~ 1.0 미만: **Bronze**
        *   `1.0 SOL` 이상 ~ 3.0 미만: **Silver**
        *   `3.0 SOL` 이상 ~ 5.0 미만: **Gold**
        *   `5.0 SOL` 이상 ~ 10.0 미만: **Platinum**
        *   `10.0 SOL` 이상: **Diamond**
*   **프론트엔드 (`web/app/launch/page.tsx`):**
    *   런칭 페이지의 담보금 입력란을 '직접 입력'과 더불어 `[0.1] [0.5] [1] [3] [5] [10]` 버튼 형태의 프리셋으로 제공합니다.
    *   마감시간 선택 드롭다운(1시간, 6시간, 12시간, 24시간 등)을 추가합니다.

## 2. 졸업(Graduation) 및 환불(Refund) 로직의 완벽한 분리

"러그풀"이라는 주관적 단어를 시스템에서 완전히 삭제하고, 오직 **"졸업(Graduation) 여부"**와 **"마감시간 초과 여부"**만으로 담보금의 향방을 결정합니다.

### 🛠️ 온체인 로직 수정 지침
*   **`release_escrow` (배포자의 담보금 회수 - `escrow.rs`):**
    *   기존의 `24시간 경과` 조건을 **삭제**합니다.
    *   대신 **`require!(curve.graduated == true)`** 조건을 추가합니다.
    *   *효과:* 24시간이 안 지났어도, 1시간 만에 졸업시켰다면 배포자는 즉시 담보금을 돌려받을 수 있습니다. (엄청난 동기 부여)
*   **`process_refund` (유저의 환불/보증금 분배 로직 - `refund.rs`):**
    *   기존의 `escrow.rugged == true` 조건을 **삭제**합니다.
    *   대신 **`require!(curve.graduated == false)`** 와 **`require!(now >= escrow.deadline_timestamp)`** 조건을 추가합니다.
    *   *효과:* 관리자(Authority)가 러그풀 마킹을 할 필요 없이, 데브가 지정한 마감시간이 지났는데 졸업을 못 한 모든 토큰의 담보금은 유저들이 알아서 클레임(Claim)해 갈 수 있습니다.
*   **`mark_rugged` 인스트럭션:**
    *   더 이상 필요 없으므로 삭제하거나 Deprecated 처리합니다.

## 3. 프레셔 스코어 (Pressure Score) 및 전광판 UI (Marquee)

짧은 시간 안에 많은 돈이 걸려 있을수록 도파민이 폭발합니다. 이를 수치화한 **Pressure Score(압박 점수)**를 도입하고, 웹 UI를 역동적으로 개편합니다.

### 🧮 Pressure Score 계산식
*   **공식:** `(에스크로 SOL / max(남은 시간(초), 1)) * 100`
*   *예시:* 10 SOL이 걸려있고 마감까지 60초 남았다면? Score = (10 / 60) * 100 = **16.6점**
*   *특징:* 시간이 줄어들수록(분모가 작아질수록) 점수가 기하급수적으로 폭발하여 유저들의 FOMO를 자극합니다.

### 🚀 서버 부하를 고려한 아키텍처 (Agent Action - Critical)
시간(초)이 지날 때마다 점수가 바뀌기 때문에, **백엔드(DB)에서 매초 점수를 계산해서 내려주면 서버가 터집니다.** 따라서 연산을 프론트엔드로 위임(Offload)해야 합니다.

1.  **백엔드 (API & DB):**
    *   백엔드는 절대 Pressure Score를 직접 계산해서 저장하지 않습니다.
    *   API는 토큰 데이터와 함께 **`escrowSol` (담보금)** 과 **`deadlineTimestamp` (마감 유닉스 타임)** 두 가지 정적 데이터만 프론트엔드에 전달합니다.
2.  **프론트엔드 (Client-side Calculation):**
    *   브라우저의 `requestAnimationFrame` 또는 `setInterval` (1초 단위)을 사용하여 클라이언트 단에서 실시간으로 Pressure Score를 계산하고 UI를 업데이트합니다.
    *   **정렬(Sorting):** `/bounty` 페이지에서 리스트를 정렬할 때도, 백엔드에서 받아온 전체 리스트(또는 상위 N개)를 프론트엔드의 메모리 상에서 1초마다 재정렬(Re-sorting)하여 화면에 렌더링합니다. (React의 `useMemo`와 `setInterval` 활용)

### 🖥️ 프론트엔드 UI/UX 구현 지침 (`web/`)
1.  **상단 전광판 (Top Marquee - 좌에서 우로 흐름):**
    *   **🔥 DEADLINE APPROACHING:** 마감 시간이 1시간 이내로 남은 토큰들을 남은 시간 순(오름차순)으로 보여줍니다. (예: `$DOGE - 00:15:32 남음!`)
2.  **하단 전광판 (Bottom Marquee - 우에서 좌로 흐름):**
    *   **💎 TOP BOUNTY (에스크로 높은 순):** 담보금이 가장 많이 걸린 토큰들을 순위와 함께 보여줍니다.
3.  **신규 페이지 (`/bounty` 또는 `/pressure`):**
    *   클라이언트 단에서 계산된 **Pressure Score**가 가장 높은 순서대로 실시간 정렬되는 하드코어 사냥꾼(Bounty Hunter) 전용 페이지입니다.

## 4. 수수료 클레임 점진적 잠금 해제 (Progressive Fee Unlock)

배포자가 워시 트레이딩(자전거래)으로 수수료만 빼먹고 도망가는 것을 방지하기 위해, **"본딩 커브가 도달한 최대 진행률(Max Progress)"**에 비례해서만 수수료를 클레임할 수 있게 만듭니다.

### 🧮 수학적 공식 (Math)
*   `해제된 수수료` = `(총 누적 수수료) * (역대 최대 모인 SOL / 졸업 기준 SOL)`
*   *예시:* 졸업 기준이 85 SOL인데, 커브가 한때 42.5 SOL(50%)까지 올라갔다가 덤핑으로 10 SOL로 떨어졌다면? 배포자는 총 누적 수수료의 **50%**까지만 클레임할 수 있습니다.

### 🛠️ 온체인 스마트 컨트랙트 수정 지침 (`state.rs`, `protocol.rs`)
*   **상태 구조체 변경 (`BondingCurve`):**
    *   기존 `claimable_fees` 필드를 삭제하고 아래 3개 필드로 대체/추가합니다.
        1.  `max_reserve_reached: u64` (역대 최대 리저브 잔액)
        2.  `total_deployer_fees: u64` (배포자 몫으로 할당된 총 누적 수수료)
        3.  `claimed_deployer_fees: u64` (배포자가 이미 클레임해서 가져간 수수료)
*   **`buy_tokens` 로직 수정 (`bonding_curve.rs`):**
    *   매수 시 리저브가 증가하므로, `max_reserve_reached = max(max_reserve_reached, reserve_balance)` 로 갱신합니다. (단, `GRADUATION_THRESHOLD`를 초과하지 않도록 `min` 처리)
    *   발생한 배포자 수수료를 `total_deployer_fees`에 누적합니다.
*   **`sell_tokens` 로직 수정 (`bonding_curve.rs`):**
    *   매도 시 리저브는 감소하지만 `max_reserve_reached`는 깎지 않습니다. (한 번 도달한 고점은 유지)
    *   발생한 배포자 수수료를 `total_deployer_fees`에 누적합니다.
*   **`claim_fees` 로직 전면 수정 (`protocol.rs`):**
    *   `unlocked_fees = (total_deployer_fees as u128 * max_reserve_reached as u128) / GRADUATION_THRESHOLD as u128`
    *   `amount_to_claim = unlocked_fees - claimed_deployer_fees`
    *   `require!(amount_to_claim > 0)`
    *   `claimed_deployer_fees += amount_to_claim`
    *   계산된 `amount_to_claim`만큼 배포자에게 SOL을 전송합니다.

## 5. 🚨 [CRITICAL] 온체인 데이터 동기화 아키텍처 (DB vs On-chain)

기존 FYRST 백엔드(`service/src/routes/trade.ts`)는 프론트엔드가 보내주는 매수/매도 API 요청을 그대로 믿고 DB에 저장하는 치명적인 결함(Client-authoritative)이 있었습니다. 
이 방식은 **봇(Bot)이 프론트엔드를 거치지 않고 스마트 컨트랙트(풀)로 직접 거래할 경우 DB와 온체인 데이터가 완전히 어긋나버리는 대참사**를 초래합니다.

이를 해결하기 위해 백엔드 아키텍처를 **온체인 이벤트 리스닝(On-chain Event Indexing)** 방식으로 전면 개편해야 합니다.

### 🛠️ 백엔드 동기화 로직 수정 지침 (Agent Action)
1.  **프론트엔드 의존성 제거:** `/api/trade` POST 엔드포인트에서 자체적으로 가격을 계산하고 DB에 넣는 로직을 전면 삭제합니다.
2.  **온체인 이벤트 리스닝 (WebSocket / Polling):**
    *   백엔드 서버가 켜질 때 Solana RPC의 WebSocket(`connection.onLogs` 또는 `connection.onProgramAccountChange`)을 연결하거나, **Helius Webhook**을 사용하여 FYRST 프로그램(PROGRAM_ID)에서 발생하는 모든 트랜잭션을 감시합니다.
    *   누군가(유저든 봇이든) 매수/매도를 하면, 백엔드가 이벤트를 캐치하여 트랜잭션을 파싱하고 DB(`Trade` 테이블)를 업데이트합니다.
3.  **소켓 업데이트:** DB 업데이트가 완료되면 Socket.io를 통해 프론트엔드 차트에 실시간으로 쏴줍니다.

## 6. 🚨 [CRITICAL] 졸업 이후(Post-Graduation) 자체 차트 유지 및 E2E 아키텍처

타 플랫폼(DexScreener 등)의 위젯이나 워터마크가 박힌 차트를 띄우는 것은 FYRST의 브랜드 경험(Anti-Casino, Arcade)을 크게 훼손합니다. **졸업 이전(본딩 커브)과 졸업 이후(Raydium 풀) 모두 동일한 Lightweight Charts UI를 유지**해야 합니다.

### 📊 6.1. 자체 차트 유지를 위한 데이터 파이프라인 (Data Pipeline)
차트 UI는 그대로 두되, **차트를 그리는 '데이터 소스(Data Source)'만 스위칭**하는 아키텍처를 구축합니다.

1.  **졸업 이전 (본딩 커브 기간):**
    *   *데이터 소스:* FYRST 백엔드 DB (`Trade` 테이블).
    *   *방식:* 기존처럼 온체인 리스너가 FYRST 스마트 컨트랙트의 매매를 감지하여 DB에 넣고 소켓으로 뿌려줍니다.
2.  **졸업 이후 (Raydium 기간):**
    *   *데이터 소스:* **Birdeye API 또는 Jupiter Price API + Helius Webhook (Raydium 타겟).**
    *   *백엔드 아키텍처 변경:* 토큰이 졸업하면, 백엔드는 해당 토큰의 Raydium 풀 주소(Pool Address)를 계산/저장합니다. 이후 백엔드 리스너에 해당 Raydium 풀 주소를 추가로 구독(Subscribe)시킵니다.
    *   *데이터 정규화:* Raydium 풀에서 발생한 스왑(Swap) 이벤트를 백엔드가 수신하면, 이를 FYRST DB의 `Trade` 테이블 형식에 맞게 정규화(Normalize)하여 저장합니다.
    *   *결과:* 프론트엔드는 데이터 소스가 바뀐 줄도 모른 채, 평소처럼 백엔드 API(`/api/trade`)와 소켓을 통해 데이터를 받아 **기존과 똑같은 예쁜 차트(Lightweight Charts)**를 계속 그려냅니다.

### 🖥️ 6.2. 프론트엔드 UI 상태 전환 (State Transition)
차트는 그대로 유지되지만, **매매(Trade) 버튼의 동작 방식**은 완전히 바뀌어야 합니다.

1.  **매수/매도 버튼 (Trade Interface):**
    *   `graduated = true` 상태를 감지하면, 기존의 FYRST `buy_tokens` / `sell_tokens` 스마트 컨트랙트 호출 로직을 **비활성화**합니다.
    *   대신 **Jupiter API (또는 Raydium SDK)**를 호출하여 Raydium 풀에서 스왑(Swap)하는 로직으로 버튼의 동작을 백그라운드에서 교체(Switch)합니다.
    *   유저 입장에서는 차트도 똑같고 버튼도 똑같이 생겼지만, 내부적으로는 Raydium과 통신하게 됩니다.
2.  **데브 전용 대시보드 (Deployer Panel):**
    *   **[ 💰 수수료 클레임 (Claim Fees) ]** 버튼 활성화 -> 온체인 `claim_fees` 호출.
    *   **[ 🔓 에스크로 회수 (Release Escrow) ]** 버튼 활성화 -> 온체인 `release_escrow` 호출.

### 🧪 6.3. 로컬 포크 환경에서의 완벽한 E2E 테스트 시나리오 (Agent Action)
에이전트는 프론트엔드와 백엔드를 연동하여 다음 시나리오를 로컬에서 직접 클릭하며 테스트할 수 있도록 환경을 구축해야 합니다.

1.  **환경 셋업:** `solana-test-validator --url mainnet-beta`, 백엔드 `LOCAL_TEST_MODE=true` (RPC 리스너 가동).
2.  **런칭 (UI):** 웹에서 "5 SOL 담보, 1시간 마감"으로 토큰 런칭.
3.  **졸업 펌핑 (UI):** 웹에서 85 SOL 어치를 한 번에 매수.
4.  **마이그레이션 확인:** 트랜잭션 성공 후 UI가 "🚀 Raydium 이전 완료" 상태로 자동 전환되는지 확인. (차트 UI는 그대로 유지됨)
5.  **Raydium 매매 테스트 (UI):** 전환된 UI에서 1 SOL 어치를 매수해봄.
    *   *주의:* 로컬에서는 Jupiter API가 신규 풀을 인식하지 못하므로, `@raydium-io/raydium-sdk-v2`를 사용하여 직접 CPMM 풀에 Swap 트랜잭션을 꽂아넣는 폴백(Fallback) 로직이 작동해야 합니다.
    *   매수 성공 시, 로컬 백엔드 RPC 리스너가 Raydium 스왑 트랜잭션을 감지하고 DB를 업데이트하여 **자체 차트에 캔들이 새로 그려지는지** 확인합니다.
6.  **데브 보상 테스트 (UI):** 데브 지갑으로 연결하여 `Claim Fees`와 `Release Escrow` 버튼을 눌러 SOL이 정확히 들어오는지 확인.

## 7. 개발 에이전트 주의사항 (Critical Warnings)

1.  **Zero Division 방지:** 프론트엔드에서 Pressure Score 계산 시 남은 시간이 0 이하가 될 경우 `Math.max(남은 초, 1)` 처리를 반드시 해야 합니다.
2.  **Rust Math Overflow:** 온체인에서 `unlocked_fees` 계산 시 `total_deployer_fees * max_reserve_reached`가 `u64` 범위를 초과할 수 있으므로 반드시 `u128`로 캐스팅하여 계산 후 `u64`로 다운캐스팅해야 합니다.
3.  **졸업(Graduation) 상태 동기화:** 온체인에서 `curve.graduated = true`가 되는 순간, 백엔드 DB(`Token` 모델)와 프론트엔드 소켓(`trade:executed` 이벤트)에도 이 상태가 즉시 반영되어 UI에 폭죽이 터지거나 상태가 변경되도록 연결 고리를 확인하세요.
4.  **에스크로 유실 방지:** `release_escrow` 호출 시 에스크로 PDA가 닫히면서(Close) 렌트비(Rent)까지 데브에게 돌아가도록 `close = deployer` 속성을 Anchor에 추가하세요.
