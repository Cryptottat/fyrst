# FYRST 솔라나 밈코인 메타 대응 전략 (최종 개발 지침서 v9 - 아키텍처 및 테스트 전략 확정)

> **문서 목적:** AI 에이전트가 이 문서를 읽고 즉시 온체인 프로그램(Rust), 백엔드, 프론트엔드를 에러 없이 안전하게 수정할 수 있도록 구체적인 구현 로직과 수학적 공식을 정의합니다. 
> **🚨 핵심 업데이트:** 가장 안정적인 **Helius Webhook 기반의 온체인 데이터 동기화**를 도입하고, 실무 최적화된 **Devnet + Local Fork 투트랙 테스트 전략**을 확정합니다.

---

## 1. 에스크로 담보금 및 마감시간(Deadline) 자율화

배포자(Dev)가 런칭 시 담보금과 **'졸업 마감시간(Deadline)'**을 스스로 선택하게 합니다.

### 🛠️ 온체인 스마트 컨트랙트 수정 지침 (`programs/fyrst/src/`)
*   **상태 구조체 변경 (`state.rs`의 `EscrowVault`):**
    *   기존 `created_at: i64` 외에 **`deadline_timestamp: i64`** 필드를 추가합니다.
*   **`create_escrow` 인스트럭션 (`escrow.rs`):**
    *   클라이언트로부터 `duration_seconds` (예: 1시간=3600)를 인자로 받아 `escrow.deadline_timestamp = Clock::get()?.unix_timestamp + duration_seconds;` 로 저장합니다.
*   **담보금 하한선 수정 (`constants.rs`):**
    *   `MIN_COLLATERAL`을 1 SOL에서 **0.1 SOL (100,000,000 lamports)** 로 하향 조정합니다.

---

## 2. 졸업(Graduation) 및 환불(Refund) 로직의 완벽한 분리

오직 **"졸업 여부"**와 **"마감시간 초과 여부"**만으로 담보금의 향방을 결정합니다.

### 🛠️ 온체인 로직 수정 지침
*   **`release_escrow` (배포자의 담보금 회수 - `escrow.rs`):**
    *   조건: **`require!(curve.graduated == true)`** (졸업 성공 시 즉시 회수 가능)
*   **`process_refund` (유저의 보증금 사냥 - `refund.rs`):**
    *   조건: **`require!(curve.graduated == false)`** AND **`require!(now >= escrow.deadline_timestamp)`** (마감시간 종료 및 졸업 실패 시 유저들이 N빵)

## 3. 프레셔 스코어 (Pressure Score) 및 전광판 UI (Marquee)

### 🧮 Pressure Score 계산식
*   **공식:** `(에스크로 SOL / max(남은 시간(초), 1)) * 100`

### 🚀 서버 부하를 고려한 아키텍처
1.  **백엔드:** 점수를 계산하지 않고 `escrowSol`과 `deadlineTimestamp`만 내려줍니다.
2.  **프론트엔드:** 브라우저에서 `setInterval`을 이용해 1초마다 남은 시간을 구하고 점수를 계산하여 화면을 재정렬(Re-sorting)합니다.

---

## 4. 수수료 클레임 점진적 잠금 해제 (Progressive Fee Unlock)

### 🛠️ 온체인 스마트 컨트랙트 수정 지침 (`state.rs`, `protocol.rs`)
*   **상태 구조체 변경 (`BondingCurve`):**
    *   `max_reserve_reached: u64`, `total_deployer_fees: u64`, `claimed_deployer_fees: u64` 필드 도입.
*   **`claim_fees` 로직:**
    *   `unlocked_fees = (total_deployer_fees * max_reserve_reached) / GRADUATION_THRESHOLD`
    *   `amount_to_claim = unlocked_fees - claimed_deployer_fees` (Zero Division 및 Overflow 주의: `u128` 캐스팅 필수)

## 5. 🚨 [CRITICAL] Helius Webhook 기반 온체인 데이터 동기화

프론트엔드의 API 호출에 의존하는 기존 방식(DB Mode)은 봇(Bot) 거래를 누락시키는 치명적 결함이 있습니다. 이를 해결하기 위해 업계 표준인 **Helius Webhook**을 사용하여 백엔드를 구축합니다.

### 🛠️ 백엔드 아키텍처 수정 지침 (Agent Action)
1.  **프론트엔드 의존성 제거:** `/api/trade` POST 엔드포인트에서 자체적으로 가격을 계산하고 DB에 넣는 로직을 전면 삭제합니다.
2.  **Helius Webhook 엔드포인트 생성 (`/api/webhooks/helius`):**
    *   Helius 대시보드에서 FYRST의 `PROGRAM_ID`에서 발생하는 모든 트랜잭션을 이 엔드포인트로 쏘도록 설정합니다.
    *   백엔드는 Helius가 보내주는 파싱된 트랜잭션 데이터(또는 Raw 데이터 파싱)를 분석하여 **런칭, 매수, 매도, 졸업 이벤트**를 식별합니다.
3.  **DB 및 소켓 업데이트:**
    *   웹훅을 통해 확인된 정확한 온체인 데이터(`tokenAmount`, `solAmount`, `newSupply` 등)를 바탕으로 DB(`Trade`, `Token` 테이블)를 업데이트합니다.
    *   업데이트가 완료되면 Socket.io를 통해 프론트엔드에 `trade:executed` 및 `price:update` 이벤트를 브로드캐스팅합니다.

## 6. 투트랙(Two-Track) 테스트 전략 확정

Helius Webhook과 Raydium 마이그레이션이라는 두 가지 큰 산을 넘기 위해, 테스트 환경을 목적에 맞게 분리합니다.

### 🧪 Track 1: Devnet (핵심 로직 & Helius 웹훅 테스트)
Helius는 Devnet 환경을 완벽하게 지원합니다. 따라서 매매 인덱싱 파이프라인은 무조건 Devnet에서 테스트해야 합니다.
*   **목적:** 토큰 런칭, 본딩 커브 매수/매도, Helius Webhook을 통한 DB 자동 동기화, 프레셔 스코어 UI 동작 확인.
*   **방법:** 
    1. FYRST 프로그램을 Devnet에 배포합니다.
    2. 로컬 백엔드 서버를 `ngrok` 등을 이용해 외부로 노출시킵니다.
    3. Helius Devnet 대시보드에서 Webhook URL을 내 `ngrok` 주소로 설정하고 테스트를 진행합니다.

### 🌉 Track 2: Local Mainnet Fork (Raydium 졸업 테스트)
Devnet의 Raydium은 잦은 오류와 버전 문제로 테스트가 불가능에 가깝습니다. 졸업 이후의 유동성 풀 생성은 메인넷 포크 환경에서 진행합니다.
*   **목적:** 85 SOL 달성 시 거래 중단, 에스크로 회수, Raydium CPMM 유동성 풀 생성 및 LP 토큰 소각(Burn) 등 졸업(Graduation) 프로세스 완벽 검증.
*   **방법:**
    ```bash
    # 메인넷 RPC URL을 사용하여 로컬 검증자를 포크 모드로 실행
    solana-test-validator --url https://api.mainnet-beta.solana.com --reset
    ```
    *   이 환경에서는 Helius 웹훅이 동작하지 않으므로, 백엔드의 매매 기록 로직은 잠시 수동(Mock)으로 돌리거나, CLI 스크립트(TS/Rust)를 짜서 85 SOL을 한 번에 매수하여 강제로 졸업 상태를 만든 뒤 Raydium 연동만 집중적으로 테스트합니다.
