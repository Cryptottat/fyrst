# FYRST 솔라나 밈코인 메타 대응 전략 (최종 개발 지침서 v8 - 아키텍처 전면 수정)

> **문서 목적:** AI 에이전트가 이 문서를 읽고 즉시 온체인 프로그램(Rust), 백엔드, 프론트엔드를 에러 없이 안전하게 수정할 수 있도록 구체적인 구현 로직과 수학적 공식을 정의합니다. 
> **🚨 중요 업데이트:** 프론트엔드 API 호출에 의존하는 기존 매매 기록 방식의 치명적 결함을 수정하고, **온체인 이벤트 리스닝(On-chain Event Indexing)** 기반의 견고한 아키텍처를 도입합니다.

---

## 1. 에스크로 담보금 및 마감시간(Deadline) 자율화

배포자(Dev)가 런칭 시 담보금과 **'졸업 마감시간(Deadline)'**을 스스로 선택하게 합니다.

### 🛠️ 온체인 스마트 컨트랙트 수정 지침 (`programs/fyrst/src/`)
*   **상태 구조체 변경 (`state.rs`의 `EscrowVault`):**
    *   기존 `created_at: i64` 외에 **`deadline_timestamp: i64`** 필드를 추가합니다.
*   **`create_escrow` 인스트럭션 (`escrow.rs`):**
    *   클라이언트로부터 `duration_seconds` (예: 1시간=3600)를 인자로 받아 `escrow.deadline_timestamp = Clock::get()?.unix_timestamp + duration_seconds;` 로 저장합니다.
*   **담보금 하한선 수정 (`constants.rs`):**
    *   `MIN_COLLATERAL`을 1 SOL에서 **0.1 SOL (100,000,000 lamports)** 로 하향 조정합니다.

---

## 2. 졸업(Graduation) 및 환불(Refund) 로직의 완벽한 분리

오직 **"졸업 여부"**와 **"마감시간 초과 여부"**만으로 담보금의 향방을 결정합니다.

### 🛠️ 온체인 로직 수정 지침
*   **`release_escrow` (배포자의 담보금 회수 - `escrow.rs`):**
    *   조건: **`require!(curve.graduated == true)`** (졸업 성공 시 즉시 회수 가능)
*   **`process_refund` (유저의 보증금 사냥 - `refund.rs`):**
    *   조건: **`require!(curve.graduated == false)`** AND **`require!(now >= escrow.deadline_timestamp)`** (마감시간 종료 및 졸업 실패 시 유저들이 N빵)

## 3. 프레셔 스코어 (Pressure Score) 및 전광판 UI (Marquee)

### 🧮 Pressure Score 계산식
*   **공식:** `(에스크로 SOL / max(남은 시간(초), 1)) * 100`

### 🚀 서버 부하를 고려한 아키텍처
1.  **백엔드:** 점수를 계산하지 않고 `escrowSol`과 `deadlineTimestamp`만 내려줍니다.
2.  **프론트엔드:** 브라우저에서 `setInterval`을 이용해 1초마다 남은 시간을 구하고 점수를 계산하여 화면을 재정렬(Re-sorting)합니다.

---

## 4. 수수료 클레임 점진적 잠금 해제 (Progressive Fee Unlock)

### 🛠️ 온체인 스마트 컨트랙트 수정 지침 (`state.rs`, `protocol.rs`)
*   **상태 구조체 변경 (`BondingCurve`):**
    *   `max_reserve_reached: u64`, `total_deployer_fees: u64`, `claimed_deployer_fees: u64` 필드 도입.
*   **`claim_fees` 로직:**
    *   `unlocked_fees = (total_deployer_fees * max_reserve_reached) / GRADUATION_THRESHOLD`
    *   `amount_to_claim = unlocked_fees - claimed_deployer_fees` (Zero Division 및 Overflow 주의: `u128` 캐스팅 필수)

## 5. 🚨 [CRITICAL] 온체인 매매 데이터 동기화 아키텍처 (DB vs On-chain)

기존 FYRST 백엔드(`service/src/routes/trade.ts`)는 프론트엔드가 보내주는 매수/매도 API 요청을 그대로 믿고 DB에 저장하는 치명적인 결함(Client-authoritative)이 있었습니다. 
이 방식은 **봇(Bot)이 프론트엔드를 거치지 않고 스마트 컨트랙트(풀)로 직접 거래할 경우 DB와 온체인 데이터가 완전히 어긋나버리는 대참사**를 초래합니다.

이를 해결하기 위해 백엔드 아키텍처를 **온체인 데이터 검증 및 리스닝 방식**으로 전면 개편해야 합니다.

### 🛠️ 백엔드 동기화 로직 수정 지침 (Agent Action)

**방법 A: 트랜잭션 사후 검증 (Transaction Verification)**
1.  프론트엔드가 `/api/trade`에 `txSignature`를 보냅니다.
2.  백엔드는 이 요청을 바로 DB에 넣지 않고, **Solana RPC(`connection.getTransaction`)를 호출하여 해당 트랜잭션이 실제로 성공했는지, FYRST 프로그램과 상호작용했는지, 실제 매수/매도 금액이 얼마인지 온체인 데이터를 파싱하여 검증**한 뒤 DB에 저장합니다.

**방법 B: 온체인 이벤트 리스닝 (WebSocket / Polling) - 권장**
1.  백엔드 서버가 켜질 때 Solana RPC의 WebSocket(`connection.onLogs` 또는 `connection.onProgramAccountChange`)을 연결하여 FYRST 프로그램(PROGRAM_ID)에서 발생하는 모든 트랜잭션을 감시합니다.
2.  누군가(유저든 봇이든) 매수/매도를 하면, 백엔드가 이벤트를 캐치하여 트랜잭션을 파싱하고 DB(`Trade` 테이블)를 업데이트합니다.
3.  DB 업데이트가 완료되면 Socket.io를 통해 프론트엔드 차트에 실시간으로 쏴줍니다.

## 6. 로컬 테스트 환경 (Mainnet Fork) 및 RPC 세팅

Helius Webhook은 로컬 환경(127.0.0.1)의 트랜잭션을 감지할 수 없습니다. 따라서 완벽한 로컬 테스트를 위해서는 **Solana Native RPC**를 활용해야 합니다.

### 🚀 로컬 포크 환경 세팅
```bash
# 1. 메인넷 RPC URL을 사용하여 로컬 검증자(Validator)를 포크 모드로 실행
solana-test-validator --url https://api.mainnet-beta.solana.com --reset

# 2. 내 로컬 지갑에 10,000 SOL 에어드랍
solana airdrop 10000

# 3. 로컬 환경에 FYRST 프로그램 배포
anchor deploy --provider.cluster localnet
```

### 🧪 테스트 시 백엔드 환경변수 세팅
*   백엔드 `.env`의 `SOLANA_RPC_URL`을 `http://127.0.0.1:8899`로 설정합니다.
*   백엔드는 이 로컬 RPC를 통해 `getTransaction`을 호출하거나 `onLogs` 웹소켓(`ws://127.0.0.1:8900`)을 연결하여, 봇이 치는 거래나 직접 치는 거래 모두 완벽하게 DB와 동기화되는지 테스트합니다.
*   이 구조가 완성되어야 추후 Raydium 풀로 이전된 후의 거래나, 본딩 커브 상에서의 봇 거래를 누락 없이 추적할 수 있습니다.
