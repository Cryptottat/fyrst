# FYRST 솔라나 밈코인 메타 대응 전략 (최종 개발 지침서 v11 - 졸업 이후 E2E 완벽 가이드)

> **문서 목적:** AI 에이전트가 이 문서를 읽고 즉시 온체인 프로그램(Rust), 백엔드, 프론트엔드를 에러 없이 안전하게 수정할 수 있도록 구체적인 구현 로직을 정의합니다. 
> **🚨 핵심 업데이트:** 유저님의 지적을 반영하여, **졸업 이후(Post-Graduation) Raydium 연동, 매매, 수수료 클레임, 에스크로 회수까지 프론트엔드 UI에서 완벽하게 E2E(End-to-End) 테스트**를 수행하기 위한 아키텍처를 추가했습니다.

---

## 1. 에스크로 담보금 및 마감시간(Deadline) 자율화
*   `EscrowVault`에 `deadline_timestamp: i64` 추가. `MIN_COLLATERAL` = 0.1 SOL.

## 2. 졸업(Graduation) 및 환불(Refund) 로직의 완벽한 분리
*   `release_escrow`: `require!(curve.graduated == true)`
*   `process_refund`: `require!(curve.graduated == false)` AND `require!(now >= escrow.deadline_timestamp)`

## 3. 프레셔 스코어 (Pressure Score) 및 전광판 UI
*   `Pressure Score = (에스크로 SOL / max(남은 시간(초), 1)) * 100` (프론트엔드 실시간 연산)

## 4. 수수료 클레임 점진적 잠금 해제 (Progressive Fee Unlock)
*   `unlocked_fees = (total_deployer_fees * max_reserve_reached) / GRADUATION_THRESHOLD`

---

## 5. 🚨 [CRITICAL] 졸업 이후(Post-Graduation) E2E 테스트 및 UI/UX 아키텍처

졸업을 달성하는 것에서 끝나는 것이 아니라, **졸업 이후 웹사이트에서 Raydium 풀을 통해 매매가 되는지, 데브가 돈을 잘 찾아가는지** UI 단에서 완벽하게 테스트되어야 합니다.

### 🛑 5.1. 수수료 축적의 오해와 진실 (기획 확정)
*   **본딩 커브 기간:** 유저들이 매매할 때마다 1%의 수수료가 발생하며, 이 중 절반이 데브 몫(`total_deployer_fees`)으로 스마트 컨트랙트에 축적됩니다.
*   **Raydium 졸업 이후:** 유동성이 Raydium으로 넘어가면, 이후 발생하는 거래 수수료는 **Raydium 유동성 공급자(LP)**에게 갑니다. FYRST는 LP 토큰을 영구 소각(Burn)하므로, **졸업 이후에는 FYRST 프로토콜이나 데브에게 추가적인 거래 수수료가 쌓이지 않습니다.** (pump.fun과 동일한 구조)
*   **결론:** 데브는 졸업하는 순간까지 쌓인 수수료의 100%를 `claim_fees` 버튼을 눌러 한 번에(또는 점진적으로) 가져가게 됩니다.

### 🖥️ 5.2. 프론트엔드 UI 상태 전환 (State Transition)
토큰이 85 SOL을 달성하여 `graduated = true`가 되는 순간, 프론트엔드 UI는 다음과 같이 완전히 변신해야 합니다.

1.  **차트 (Chart):**
    *   *프로덕션(Mainnet):* DexScreener 또는 GeckoTerminal API의 iframe/위젯으로 교체.
    *   *로컬 테스트(Localnet):* 외부 API는 로컬넷을 읽지 못하므로, 차트 영역에 **"로컬 테스트 모드: Raydium 풀 생성 완료"**라는 더미(Mock) 화면을 띄워 에러를 방지합니다.
2.  **매수/매도 버튼 (Trade Interface):**
    *   기존의 FYRST `buy_tokens` / `sell_tokens` 스마트 컨트랙트 호출 로직을 **비활성화**합니다.
    *   대신 **Raydium SDK (또는 Jupiter API)**를 호출하여 방금 생성된 Raydium 풀에서 스왑(Swap)하는 로직으로 버튼의 동작을 완전히 교체(Switch)합니다.
3.  **데브 전용 대시보드 (Deployer Panel):**
    *   **[ 💰 수수료 클레임 (Claim Fees) ]** 버튼 활성화 -> 클릭 시 온체인 `claim_fees` 호출 및 지갑 잔고 증가 확인.
    *   **[ 🔓 에스크로 회수 (Release Escrow) ]** 버튼 활성화 -> 클릭 시 온체인 `release_escrow` 호출 및 담보금(예: 5 SOL) 회수 확인.

### 🧪 5.3. 로컬 포크 환경에서의 완벽한 E2E 테스트 시나리오 (Agent Action)
에이전트는 프론트엔드와 백엔드를 연동하여 다음 시나리오를 로컬에서 직접 클릭하며 테스트할 수 있도록 환경을 구축해야 합니다.

1.  **환경 셋업:** `solana-test-validator --url mainnet-beta`, 백엔드 `LOCAL_TEST_MODE=true` (RPC 리스너 가동).
2.  **런칭 (UI):** 웹에서 "5 SOL 담보, 1시간 마감"으로 토큰 런칭.
3.  **졸업 펌핑 (UI):** 웹에서 85 SOL 어치를 한 번에 매수 (또는 여러 번 매수하여 수수료 축적 확인).
4.  **마이그레이션 확인:** 트랜잭션 성공 후 UI가 "🚀 Raydium 이전 완료" 상태로 자동 전환되는지 확인.
5.  **Raydium 매매 테스트 (UI):** 전환된 UI에서 1 SOL 어치를 매수해봄. (이때 프론트엔드는 내부적으로 Raydium SDK를 사용하여 로컬에 생성된 CPMM 풀과 스왑을 시도해야 하며, 성공 시 지갑에 토큰이 들어와야 함).
6.  **데브 보상 테스트 (UI):** 데브 지갑으로 연결하여 `Claim Fees`와 `Release Escrow` 버튼을 눌러 SOL이 정확히 들어오는지 확인.

### 🛠️ Raydium SDK 로컬 연동 주의사항
*   Jupiter API는 로컬넷의 신규 풀을 즉시 인덱싱하지 못합니다.
*   따라서 프론트엔드 코드를 짤 때, **"네트워크가 Localnet일 경우 Jupiter API를 거치지 않고 `@raydium-io/raydium-sdk-v2`를 사용하여 직접 CPMM 풀에 Swap 트랜잭션을 꽂아넣는 로직"**을 폴백(Fallback)으로 구현해 두어야 로컬 UI 테스트가 가능합니다.
