# Phase 11: 퍼블릭 GitHub (오픈소스 마케팅용) 구축 가이드 | new_project_guide

> 이 단계는 실제 서비스 코드(`web/`, `service/`)와는 완전히 별개로, 마케팅 및 커뮤니티 빌딩을 위해 **가짜 오픈소스 프로젝트(Public Repo)**를 생성하는 과정입니다.
> `guide for github.md`의 지침을 완벽하게 따르며, 2~6개월 간의 개발 히스토리(잔디)를 조작하여 "활발하게 개발된 프로젝트"처럼 보이게 만듭니다.

---

## 1. 기본 원칙 및 디렉토리

- **작업 폴더**: 프로젝트 루트 내의 `github/` 폴더에서만 작업합니다. (절대 `web/`이나 `service/` 코드를 이곳으로 복사하지 마십시오)
- **계정 전환 (CRITICAL)**: 푸시하기 전, 반드시 PowerShell에서 GitHub CLI(`gh`)를 사용하여 **공개용 계정(예: skaf-labs)**으로 로그인되어 있는지 확인해야 합니다. `Cryptottat` 계정으로 푸시하면 안 됩니다.
- **도구**: PowerShell 환경과 GitHub CLI(`gh`), Python(`main.py`)을 사용합니다.

---

## 2. `main.py` 스크립트 생성 및 실행

`guide for github.md`의 [Ultimate Master Prompt]를 기반으로 `github/main.py`를 작성합니다. 이 스크립트는 35개 이상의 파일과 조작된 커밋 히스토리를 자동 생성합니다.

### 필수 설정 변수 (Configuration)
- `START_DATE`: 오늘 기준 2~6개월 전 날짜
- `END_DATE`: **오늘 날짜의 하루 전** (예: 오늘이 2026년 2월 28일이면 `2026-02-27`로 설정)
- `TARGET_COMMITS`: 130개 이상

### 커밋 히스토리(잔디) 조작 규칙
- **알록달록하고 불규칙한 잔디**: 매일 똑같은 커밋 수가 아니라, 어떤 날은 많고 어떤 날은 적게(0~7개 등) 설정합니다.
- **휴일 고려**: 주말(토, 일)이나 일반적인 휴일에는 커밋 빈도를 줄이거나 없애서 실제 사람이 개발한 것처럼 자연스럽게 만듭니다.
- **커밋 내용 일치**: 모든 커밋 메시지는 중복 없이 다 달라야 하며, 해당 커밋에서 변경된 가짜 코드의 내용과 커밋 메시지가 논리적으로 일치해야 합니다.

## 3. README 및 에셋(Assets) 구성

스크립트가 생성하는 `README.md`와 에셋들은 다음을 반드시 포함해야 합니다.

1. **배너 이미지**: 프로젝트의 깃허브 배너 이미지를 생성(또는 Phase 2에서 생성된 이미지 사용)하여 `github/` 폴더 내에 포함시키고 README 상단에 배치합니다.
2. **링크 뱃지 (HTML)**: 트위터(X) 링크와 공식 웹사이트 링크를 마크다운이 아닌 **순수 HTML `<img>` 태그 뱃지**로 삽입하여 엑박을 방지합니다.
3. **구조도 (Architecture)**: Mermaid.js 문법이나 ASCII 아트를 사용한 시스템 플로우차트/구조도를 반드시 포함하여, GitHub 페이지에서 그림으로 렌더링되게 합니다.
4. **링크 무결성 검수**: README에 기재된 모든 웹사이트 URL, 트위터 링크 등이 404 에러 없이 정상적으로 연결되는지 푸시 전 꼼꼼히 체크합니다.

---

## 4. CI/CD 및 로컬 테스트 (빌드 무결성)

가짜 코드(Rust, TypeScript 등)라 할지라도 GitHub Actions(CI/CD)에서 테스트가 통과해야 "신뢰할 수 있는 프로젝트"로 보입니다.

- **로컬 빌드/테스트**: `main.py`가 코드를 생성한 후, 로컬 환경에서 해당 코드들의 빌드와 테스트가 성공하는지 반드시 확인합니다.
- `package.json`이나 `Cargo.toml`의 의존성이 완벽하게 맞물려 있어야 합니다.

---

## 5. 최종 체크리스트 및 푸시 (Push)

`main.py`를 실행하기 전과 후, 다음 체크리스트를 모두 확인하십시오.

- [ ] PowerShell에서 `gh auth status`를 쳐서 **공개용 계정(Labs)**인지 확인했는가?
- [ ] 개발 기간이 2~6개월로 설정되었고, 종료일이 어제 날짜로 잘 설정되었는가?
- [ ] 생성된 커밋 히스토리가 주말을 고려하여 불규칙하고 자연스러운가?
- [ ] 모든 커밋 메시지가 다르고, 실제 변경된 코드와 일치하는가?
- [ ] 배너 이미지가 지정된 경로에 잘 들어갔는가?
- [ ] 트위터/웹사이트 링크 뱃지가 HTML 태그로 잘 들어갔으며, 클릭 시 정상 연결되는가?
- [ ] README에 Mermaid/ASCII 구조도(플로우차트)가 잘 렌더링되는가?
- [ ] 로컬에서 가짜 코드들의 빌드 및 테스트가 모두 성공했는가? (GitHub CI/CD 패스 보장)
- [ ] 전체적으로 웹/링크 접속 시 404 에러 등 잘못된 곳이 없는지 최종 검수했는가?

모든 체크리스트가 완료되고 코드가 개선되었다면, `main.py`를 돌려 히스토리를 생성하고 **공개용 계정으로 전환된 상태**에서 `git push -u origin main`을 실행합니다.
